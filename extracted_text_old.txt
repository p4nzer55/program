400
原因：
1、源站 Web 服务器（如 Nginx、Apache）因接收到格式异常的请求（如超长 URL、非法字符、畸形 Header）而自行返回 400
2、玄武盾开启 HTTP 安全响应头配置 或 精细化访问控制 后，对请求头/参数进行校验，但该类策略默认不返回 400，而是触发拦截并返回 403 或执行挑战；
3、若客户配置了 TLS 协议版本限制（如仅允许 TLS 1.2），而客户端使用不兼容的 TLS 版本发起 HTTPS 请求，部分源站或中间件可能返回 400；
4、玄武盾支持的协议和端口范围有限，若请求使用了不支持的非标端口或协议组合（如 HTTP 协议走 443 端口），源站可能因协议协商失败返回 400。
1】请求头中 Host 字段未携带端口号（例如网站接入 HTTPS 但请求头 Host 缺少 :443），尤其在检测或防护模式下会触发校验失败；
2】客户端使用了玄武盾不支持的 HTTP 方法（默认仅放行 GET 和 POST，其余方法如 PUT、DELETE 等会被拦截）；
3】网站通过反向代理访问时，前侧代理未正确设置 Host 头，导致玄武盾校验不通过。
4】若用户在云防护平台配置站点时协议（HTTP/HTTPS）与实际访问协议不一致（如配置 HTTPS 但用 HTTP 访问），也可能返回 400。
检查：
1、是否启用了 “Host 头必须携带端口号” 校验规则（常见于微信支付回调等场景），若 Host 字段缺失端口（如 Host: example.com 而非 Host: example.com:443），可能导致源站拒绝请求并返回 400；
2、是否在站点配置中错误启用了 HTTPS-SNI 选项 但源站未正确配置 SNI 支持。
